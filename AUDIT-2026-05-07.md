# AUDIT-2026-05-07 — Quantum Badger Safety & Security Audit

## 1. Executive Summary

Quantum Badger is designed as a privacy-first, local-first AI assistant for macOS. The architecture, as described in the technical specifications, promises robust guardrails including sandboxed file access, strict web filtering, and a "Shadow Router" that intelligentally offloads tasks while preserving privacy. However, a deep audit of the current implementation reveals significant gaps between the intended security model and the actual codebase.

The most critical finding is a systemic failure in the privacy boundary: the "Shadow Router" unconditionally transmits user prompts to third-party cloud providers (Anthropic, OpenAI, or Google) for intent analysis before any policy checks or user consent. This effectively nullifies the "privacy-first" promise, as the core context of every user interaction is leaked to the cloud by default. Furthermore, the emergency "Lockdown" mode and the "Safe Mode" execution policies are not enforced in the primary execution pipelines, meaning the system continues to attempt remote operations even when explicitly forbidden.

Architecturally, the system is incomplete. Core security components such as `FileManagerService` (responsible for sandboxed access and validation) and `RateLimiter` (responsible for preventing resource exhaustion) are entirely absent. The audit logging mechanism, while providing a tamper-evident hash chain, fails to protect user privacy by logging sensitive command data in plain text.

While the codebase demonstrates high quality in terms of Swift 6 concurrency adoption and modularity, the safety posture is currently **Fragile**. The system relies on "Security by Spec" rather than "Security by Implementation." Immediate remediation is required to align the execution pipeline with the stated security policies.

## 2. P0 Issues (Critical)

### 1. Default Data Exfiltration via Shadow Router Intent Analysis
* **Why it is dangerous**: The "Shadow Router" is intended to protect privacy by choosing local inference when possible. However, its current implementation sends the user's prompt to a cloud "Mini" model to determine complexity *before* any routing decision is made. This means 100% of user queries are sent to the cloud, regardless of the user's "Local First" preference or the sensitivity of the data.
* **How it manifests**: In `ShadowRouter.performIntentAnalysis(prompt:)`, a network request is made to a cloud provider with the user's prompt embedded in the analysis instruction.
* **Concrete recommendation**: Implement a local heuristic-first or local-model-first intent analyzer. Cloud-based intent analysis must be opt-in and gated by the same consent flow as the primary inference.
* **Expected blast radius if unfixed**: Total loss of user privacy for all queries. Regulatory non-compliance (GDPR/CCPA) regarding unauthorized data transmission.

### 2. Failure to Enforce Lockdown and Execution Policies
* **Why it is dangerous**: The `SecurityPolicy` defines an `isLockdown` flag and `allowsRemoteOperations` property. However, the `AppCoordinator` and `HybridExecutionManager` do not check these flags before initiating web fetches or cloud inference. A user in "Lockdown" mode remains vulnerable to network-based attacks and data exfiltration.
* **How it manifests**: `AppCoordinator.executePipeline` and `HybridExecutionManager.executeCore` proceed with routing and remote execution without consulting the `SecurityPolicyManager`.
* **Concrete recommendation**: Inject `SecurityPolicyManager` into the execution pipeline and add an explicit `guard policy.allowsRemoteOperations else { throw ... }` check before any `WebBrowserService` or `CloudInferenceService` calls.
* **Expected blast radius if unfixed**: Users under the false impression of security (Lockdown) are actually unprotected. System compromise during high-risk scenarios.

## 3. P1 Issues (High Severity)

### 1. Missing Core Security Components (FileManagerService & RateLimiter)
* **Why it is dangerous**: The technical spec mandates `FileManagerService` for magic-number validation and sandboxing, and `RateLimiter` for preventing runaway loops and resource exhaustion. Neither exists. Current file operations (in `ModelsViewModel`) use `FileManager.default` directly, providing no protection against malicious file paths or resource-intensive operations.
* **How it manifests**: No source code exists for these classes; they are only referenced in `TECHNICAL_SPEC.md` and as placeholders in `BadgerRuntime.swift` (actually, they aren't even aliased in `BadgerRuntime.swift`, only mentioned in docs).
* **Concrete recommendation**: Implement `FileManagerService` with strict path validation and `RateLimiter` using a token-bucket algorithm as specified in the architecture.
* **Expected blast radius if unfixed**: Susceptibility to Denial of Service (DoS) via resource exhaustion. Potential for path traversal or unsafe file parsing if new features are added using raw `FileManager`.

### 2. Plain-Text PII Leakage in Audit Logs
* **Why it is dangerous**: `AuditLogService` logs `RemoteCommandReceived` events which include the first 100 characters of the user's command. If a user includes a password or API key at the start of a prompt, it is written to a persistent, unencrypted log file.
* **How it manifests**: `AppCoordinator.logCommandReceipt` passes the raw `command.prefix(100)` to the audit service.
* **Concrete recommendation**: All data sent to `AuditLogService` must be passed through the `InputSanitizer` or `PrivacyEgressFilter` first.
* **Expected blast radius if unfixed**: Persistent storage of PII on disk in an easily accessible format. Failure of "Privacy-First" audit requirements.

## 4. P2 Issues (Medium Severity)

### 1. Inconsistent Sanitization Logic
* **Why it is dangerous**: `InputSanitizer` and `PrivacyEgressFilter` have overlapping responsibilities (detecting SSNs, API keys, etc.) but use different regex patterns and different replacement strings. This creates a "gap" where one filter might miss what the other catches, leading to inconsistent protection.
* **How it manifests**: `InputSanitizer.piiPatterns` vs `PrivacyEgressFilter.matchers`.
* **Concrete recommendation**: Consolidate all PII/Sensitive detection patterns into a single shared `PrivacyRegistry` in `BadgerCore` and have both services consume the same source of truth.
* **Expected blast radius if unfixed**: Maintenance burden and higher probability of bypasses due to "shadow" regex logic.

### 2. Lack of Encryption for Persistent Audit Logs
* **Why it is dangerous**: While the logs are "tamper-evident" (via hashing), they are not "confidential." Anyone with access to the filesystem can read the audit trail.
* **How it manifests**: `AuditLogService` writes JSON strings directly to `.log` files.
* **Concrete recommendation**: Use `AES-GCM` (available in `BadgerCore.Crypto`) to encrypt log blocks before writing to disk, using a key stored in the Secure Enclave via `KeyManager`.
* **Expected blast radius if unfixed**: Leakage of system behavior and meta-data to local attackers or other applications.

---

## 5. P3 Issues (Low Severity)

### 1. Brittle Intent Analysis Parsing
* **Why it is dangerous**: The `ShadowRouter` relies on a regex-based "lenient" parser for JSON responses from the cloud. This is prone to failure if the model output format changes slightly.
* **How it manifests**: `ShadowRouter.parseLenientAnalysis` uses manual regex string matching.
* **Concrete recommendation**: Use a more robust JSON repair library or strictly enforce JSON-mode if supported by the provider.
* **Expected blast radius if unfixed**: Occasional "Intent Analysis Failed" errors causing fallback to slower local heuristics.

---

## 6. P4 Issues (Nice‑to‑Have)

### 1. Versioning Mismatch
* **Why it is dangerous**: `BadgerRuntime.swift` and `BadgerCore.swift` both define `version = "0.1.1"`, but the project is moving towards a more mature 1.0 architecture.
* **How it manifests**: Static version strings in source files.
* **Concrete recommendation**: Centralize versioning in the build system or a shared `Version.xcconfig`.
* **Expected blast radius if unfixed**: Cosmetic confusion during debugging.

## 7. Recommended Remediation Sequence

1.  **Stop the Leak (P0)**: Modify `ShadowRouter` to use local heuristics for intent analysis by default. Disable cloud-based intent analysis unless explicit consent is provided.
2.  **Enforce Boundaries (P0)**: Update `AppCoordinator` and `HybridExecutionManager` to respect `isLockdown` and `ExecutionPolicy` flags.
3.  **Sanitize Logs (P1)**: Update `AppCoordinator` to sanitize commands before logging them to the audit trail.
4.  **Fill Architectural Gaps (P1)**: Implement the missing `RateLimiter` and `FileManagerService` to provide the promised guardrails.
5.  **Consolidate Privacy (P2)**: Refactor `InputSanitizer` and `PrivacyEgressFilter` to use a unified pattern registry.

## 8. 30‑Day Hardening Plan

*   **Week 1: Privacy Perimeter**: Fix the Shadow Router leak and enforce Lockdown mode. **Owner**: Security Architect. **Outcome**: No data leaves the device in Lockdown/Safe Mode.
*   **Week 2: Audit Integrity**: Sanitize all audit logs and implement log encryption. **Owner**: Core Developer. **Outcome**: Audit logs are confidential and PII-free.
*   **Week 3: Resource Protection**: Implement the `RateLimiter` and integrate it into the `AppCoordinator` pipeline. **Owner**: Runtime Developer. **Outcome**: System is resilient against DoS and runaway loops.
*   **Week 4: File Safety**: Implement `FileManagerService` and refactor `ModelsViewModel` to use it. **Owner**: Runtime Developer. **Outcome**: All file I/O is validated and sandboxed.

---

## 9. Long‑Term Architecture Recommendations

1.  **Hardware-Attested Identity**: Use the Secure Enclave not just for keys, but to sign every audit entry, providing hardware-backed proof of the audit trail's integrity.
2.  **Capability-Based Security**: Move from a global `SecurityPolicy` to a capability-based model where the `AppCoordinator` hands out scoped tokens to runtime actors, further enforcing the principle of least privilege.
3.  **Local-Only Intent Analysis**: Invest in a dedicated, extremely lightweight local model (e.g., <100M params) specifically for intent routing to remove cloud dependency for the decision-making layer entirely.
